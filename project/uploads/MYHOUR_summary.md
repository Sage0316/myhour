# MYHOUR 정리

## 1. 한 줄 정의

MYHOUR는 시간 단위의 짧은 기록을 모아, 하루를 한 편의 영상 결과물로 되돌려주는 라이프로그 앱이다.

기록은 사진, 영상, 음성, 텍스트 등 여러 방식으로 남길 수 있고, 하루가 끝나면 기록 조각들이 하나의 흐름으로 정리되어 다시 보고 싶은 결과물로 완성된다.

---

## 2. 이 앱을 만들고 싶은 이유

MYHOUR는 단순히 기록 앱을 만들고 싶어서 나온 아이디어가 아니라, 실제로 반복해서 겪은 불편과 욕구에서 시작됐다.

- 셋 로그처럼 하루를 시간 단위로 기록해보고 싶었지만 늘 실패했다.
- 정각을 맞춰 직접 타이머를 관리하는 방식은 번거로웠다.
- 사진, 영상, 음성, 텍스트처럼 다양한 방식으로 남기고 싶었다.
- 하지만 그렇게 남긴 기록은 서로 흩어져 있을 뿐, 하나의 의미 있는 결과물로 남지 않았다.
- 결국 원했던 것은 기록 자체보다, 하루가 하나의 영상처럼 남는 경험이었다.

즉 MYHOUR는 “기록하고 싶지만 계속 놓치고, 남겨도 흩어지는 하루”를 “나중에 다시 보고 웃을 수 있는 결과물”로 바꾸고 싶은 욕구에서 출발한 앱이다.

---

## 3. 문제 정의

사용자는 하루를 시간 단위로 가볍게 기록하고 싶어 하지만, 실제로는 기록 타이밍을 자주 놓치고, 어렵게 남긴 기록도 사진·영상·음성·텍스트로 흩어져 쌓일 뿐이라 다시 보게 되는 하나의 결과물로 남지 않는다.

문제를 조금 더 나누면 다음과 같다.

### 1) 기록을 지속하기 어렵다
- 정각 기록은 이상적이지만 현실에서는 자주 놓친다.
- 매번 타이머를 맞추거나 스스로 기억하는 방식은 번거롭다.

### 2) 기록이 흩어진다
- 사진은 사진첩, 영상은 갤러리, 음성은 음성메모, 글은 메모장처럼 따로 남는다.
- 기록은 남아도 하루의 흐름은 남지 않는다.

### 3) 기록의 보상이 약하다
- 남기기만 하고 다시 보지 않게 된다.
- 결과물이 없으니 지속할 동기도 약해진다.

---

## 4. 핵심 목적

MYHOUR의 목적은 단순히 하루를 저장하는 것이 아니라, 놓치기 쉬운 순간들을 부담 없이 기록하고 그것을 하루 단위의 의미 있는 결과물로 남기는 데 있다.

핵심은 “기록 앱”보다 “결과물을 돌려주는 앱”에 가깝다.

---

## 5. 핵심 경험

MYHOUR에서 가장 중요한 경험은 기록을 남기는 순간 자체보다, 하루가 끝난 뒤 또는 나중에 특정 날짜를 눌렀을 때 그날이 하나의 영상으로 돌아오는 경험이다.

예를 들면:

“어, 나 3월 1일에 뭐 했지?” 하고 눌렀을 때  
그날 생성된 영상이 나오고  
“아 이랬구나 ㅋㅋ” 하고 웃을 수 있는 경험.

즉 핵심은 업로드용 콘텐츠 제작보다도, **감정적인 만족감**이다.

- 보고 키득키득 웃을 수 있어야 한다.
- 그날의 분위기가 살아 있어야 한다.
- 다시 보고 싶은 결과물이어야 한다.

유튜브 쇼츠나 릴스 업로드는 활용처 중 하나일 뿐, 궁극적인 목적은 “심리적인 만족감이 드는 하루 회수 경험”이다.

---

## 6. 주요 사용자

MYHOUR의 핵심 사용자는 다음과 같다.

### 1) 하루를 특별하게 기록해보고 싶은 사람
단순 메모가 아니라, 자기 하루를 조금 더 의미 있게 남기고 싶은 사람.

### 2) 브이로그나 숏폼에 관심 있는 사람
직접 편집하는 건 부담스럽지만, 결과물은 영상 형태로 남기고 싶은 사람.

### 3) 자기표현 욕구가 있는 사람
일상을 단순히 저장하는 게 아니라, 나중에 다시 보고 싶고 결과물로 남기고 싶은 사람.

---

## 7. 사용자 흐름

MYHOUR의 기본 흐름은 다음과 같다.

1. 사용자가 기록 리듬을 설정한다.
   - 시작 시간
   - 종료 방식
   - 기록 간격
   - 알림 시점
   - 기록 방식

2. 앱이 설정된 시간에 맞춰 푸시를 보낸다.

3. 사용자는 알림을 받고 빠르게 기록을 남긴다.
   - 사진
   - 영상
   - 음성
   - 텍스트

4. 기록은 앱 안에 시간순으로 쌓인다.

5. 하루가 마감되면 기록 조각들이 정리된다.

6. LLM이 하루의 무드와 연출 방향을 해석한다.

7. 렌더러가 실제 영상 결과물을 만든다.

8. 사용자는 결과물을 미리 보고 저장 여부를 결정한다.

---

## 8. 핵심 기능과 그 이유

### 1) 시간대 / 간격 설정
사용자가 하루를 시간 단위로 기록하고 싶어도 실제로는 자주 놓치기 때문에 필요하다.

### 2) 알림 기능
기록을 의지에만 맡기지 않고, 앱이 적절한 시점에 기록 행동을 꺼내줘야 하기 때문에 필요하다.

### 3) 사진 / 영상 / 음성 / 텍스트 기록
하루의 순간을 항상 같은 방식으로 남기기는 어렵기 때문에, 그 순간 가장 부담 없는 방식으로 남길 수 있어야 한다.

### 4) 오늘 타임라인
흩어진 기록 조각들을 시간의 흐름 안에서 연결해 보여주기 위해 필요하다.

### 5) 하루 영상 결과물 생성
MYHOUR의 가장 중요한 목적은 하루를 하나의 결과물로 남기는 것이기 때문에 필요하다.

### 6) 결과물 미리보기 후 저장
무조건 자동 저장되기보다, 사용자가 결과물을 확인하고 저장 여부를 결정하는 방식이 더 자연스럽다.

---

## 9. LLM의 역할

MYHOUR는 AI가 없더라도 기본 구조는 성립한다.  
기록 수집, 시간순 정렬, 기본 템플릿 영상 생성은 규칙 기반으로도 가능하다.

하지만 LLM은 결과물의 **해석력과 감정적 완성도**를 높이는 역할로 들어간다.

LLM은 다음을 담당한다.

- 하루 무드 해석
- 제목 생성
- 자막 톤 결정
- 장면 연결 문구 생성
- 음악 분위기 제안
- 장면별 연출 힌트 생성
- 기록 순서 및 장면 길이 조정

즉 LLM은 영상을 직접 편집하는 것이 아니라, **그날의 결과물 톤을 정하는 해석 레이어**다.

---

## 10. 무드 타입 5가지

MYHOUR에서 무드는 심리 진단용 감정 분류가 아니라, **영상 결과물의 무드 타입**이다.

우리가 정리한 5가지 무드는 다음과 같다.

1. 잔잔함  
2. 웃김  
3. 감성  
4. 정신없음  
5. 뿌듯함

LLM은 기록물과 간단한 사용자 입력을 바탕으로 이 5개 중 하나 또는 가까운 흐름을 정한다.

---

## 11. 무드 결정 방식

영상 무드는 사용자가 처음부터 고정해서 고르는 것이 아니라, 그날의 기록을 보고 정해지는 쪽이 MYHOUR와 더 잘 맞는다.

예를 들면:

- 잔잔한 사진과 조용한 텍스트가 많으면 잔잔함
- 우당탕한 메모와 가벼운 멘트가 많으면 웃김
- 바쁘고 튀는 흐름이 많으면 정신없음

필요하면 사용자에게 아주 짧은 설문을 추가로 받을 수 있다.

예:
- 오늘 하루 기분은? (이모지 선택)
- 오늘 하루 차분함 정도는? (0~100)

LLM은 기록물 + 짧은 설문을 함께 보고 영상 무드를 정한다.

---

## 12. LLM이 앱에 넘겨주는 답안지 항목

개발 용어를 빼고, 사람말로 정리하면 LLM은 앱에 대충 이런 내용을 넘겨주면 된다.

- 날짜
- 오늘 무드
- 영상 제목
- 자막 말투
- 음악 분위기
- 마무리 문장
- 들어갈 기록 순서
- 장면별 연출 힌트
- 장면별 재생 시간

예를 들면:

- 오늘 무드: 웃김
- 제목: 우당탕하다가 커피로 겨우 살아난 날
- 자막 말투: 가볍고 장난스럽게
- 음악 분위기: 초반은 분주하고, 중간부터 경쾌하게
- 마무리 문장: 정신없었는데, 돌아보니 좀 웃기네
- 효과 포인트: 커피 사진으로 넘어갈 때 반짝이 느낌 전환
- 장면 길이: 사진 3초, 텍스트 2.5초, 영상 4초

즉, LLM은 “오늘 영상을 어떤 톤으로 기억하게 만들지”를 정리해주는 역할이다.

---

## 13. 기록 유형별 기본 재생 시간

기록물마다 결과물 안에서 기본적으로 보여지는 시간도 필요하다.

1차 기준으로 잡은 값은 다음과 같다.

- 사진: 3초
- 텍스트: 2.5초
- 영상: 4초
- 음성: 5초

여기서 LLM은 장면 중요도나 무드에 따라 조금씩 조정하는 역할만 맡는다.

예:
- 분위기 전환 포인트인 커피 사진은 3.5초
- 연결용 사진은 2초
- 마무리 영상은 4초

즉, 기본값은 앱이 갖고 있고, LLM은 필요할 때만 조정한다.

---

## 14. 저장 방식

MYHOUR는 SNOW 같은 방식처럼, 앱 안에서 먼저 결과물을 보여주고 사용자가 저장하기를 눌렀을 때 실제 저장되면 좋다는 방향으로 정리됐다.

즉 구조는 이렇다.

1. 기록은 먼저 앱 안에 쌓인다.
2. 하루가 마감되면 결과물 미리보기가 생성된다.
3. 사용자가 저장하기를 눌러야 기기에 저장된다.
4. 어떤 날은 영상 생성 없이 자료만 앱 안에 남겨둘 수도 있다.

이 방식은 사용자가 결과물을 보고 판단할 수 있어서 더 자연스럽다.

---

## 15. 하루 마감 방식

기본적으로는 사용자가 설정한 종료 시간이 되면 하루가 자동 마감되는 구조로 잡았다.

하지만 사용자는 홈에서 **오늘 하루 마감하기** 버튼을 눌러 종료 시간을 기다리지 않고도 지금 기준으로 하루를 마감할 수 있다.

즉:

- 기본 흐름: 종료 시간이 되면 자동 마감
- 예외 흐름: 사용자가 원하면 지금 바로 마감

### 종료 시간 미정 옵션
다만 생활 패턴이 일정하지 않은 사람을 위해 종료 시간 미정 옵션도 필요하다고 정리했다.

종료 시간 미정일 경우에는:

- 사용자가 직접 마감하기 전까지 기록을 계속 남길 수 있다.
- 단, 직접 마감하지 않으면 **다음날 시작 시간이 되는 순간 자동 종료**된다.

예:
- 시작 시간이 08:30이라면
- 하루를 직접 마감하지 않았을 경우
- 다음날 08:30에 자동으로 전날 하루가 종료된다.

이 구조는 늦게 자는 날, 특별한 날, 밤이 길어지는 날에도 유연하게 대응할 수 있다.

### 기록 시간 설정

  - 종료 시간 지정
  - 종료 시간 미정

- 기록 간격

  - 1시간
  - 2시간

### 알림 설정
- 1분 전
- 기록 시각

### 기록 방식 설정

- 하나로 고정
- 하나로 고정 시 기본 기록 방식 선택
  - 사진
  - 영상
  - 음성
  - 텍스트

- 기본 출력 비율
  - 9:16
  - 1:1

  - 결과물 먼저 보기
  - 자동 저장

## 17. 홈 화면 구조

홈은 “지금 기록 시작”과 “오늘 마감” 중심으로 최대한 단순하게 가는 방향으로 정리됐다.

### 홈에 들어가는 요소
- 오늘 날짜
- 짧은 한 줄 문구
- 오늘 상태 카드
  - 다음 체크인까지 남은 시간
  - 오늘 기록 개수
  - 기록 리듬
- 지금 기록하기 버튼
- 오늘 하루 마감하기 버튼
- 하단 네비게이션
  - 홈
  - 오늘
  - 아카이브
  - 설정

홈에는 오늘 보기 / 아카이브 보기 같은 중복 버튼을 두지 않고, 그 역할은 하단 네비게이션으로 넘기는 게 더 깔끔하다고 정리했다.

## 18. 페이지 구조

1차 앱 구조는 다음 네 개 탭으로 정리됐다.

### 1) 홈
- 기록 시작
- 오늘 마감
- 오늘 상태 확인

### 2) 오늘
- 오늘 남긴 기록을 시간순으로 확인
- 수정 / 삭제 / 빠진 시간 확인

### 3) 아카이브
- 날짜별 결과물 다시 보기
- 특정 날짜를 눌러 그날 영상 재생

### 4) 설정
- 기록 리듬, 알림, 기록 방식, 결과물 저장 방식 설정

## 19. 디자인 방향

디자인은 귀욤귀욤한 방향보다, **미니멀 베이스 + 소프트한 포인트**가 더 잘 맞는다고 정리됐다.

이유는 다음과 같다.

- MYHOUR의 주인공은 앱 UI보다 사용자의 하루 결과물이어야 한다.
- 너무 귀여운 톤은 취향을 많이 탈 수 있다.
- 미니멀 베이스가 결과물 영상의 감정과 무드를 더 잘 살린다.
- 다만 너무 차갑고 업무앱 같으면 안 되므로, 따뜻한 톤과 감성 포인트는 필요하다.

즉 방향은 다음처럼 잡는다.

- 전체는 차분한 미니멀
- 포인트 컬러는 부드럽고 따뜻하게
- 무드 칩, 결과물 썸네일, 영상 화면 쪽에서만 감성 포인트를 강조

## 20. 지금까지의 핵심 정리

MYHOUR는 단순 기록 앱이 아니다.  
시간 단위의 짧은 조각들을 모아, 나중에 특정 날짜를 눌렀을 때 “아 이랬구나 ㅋㅋ” 하고 웃을 수 있는 하루 영상으로 되돌려주는 앱이다.

핵심은 다음 세 가지다.

1. 기록을 쉽게 남길 수 있어야 한다.
2. 흩어진 기록을 하루의 흐름으로 묶어야 한다.
3. 결과물이 감정적으로 만족스러워야 한다.

그리고 LLM은 이 구조 안에서, 기록을 요약하는 도구가 아니라 **그날의 결과물을 어떤 톤으로 기억하게 만들지 정하는 해석 레이어**로 작동한다.
